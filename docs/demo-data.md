# Demo Data Guide

Career Tracker のポートフォリオ用スクリーンショット撮影に使うデモデータ方針と、`supabase/seed.sql` の実行手順です。

## 目的

デモデータは、以下を自然に確認・撮影できる状態を作るために用意します。

- ダッシュボードの集計
- 期限切れタスク
- 7日以内のタスク
- 優先度高の求人
- 複数ステータスに分散した応募・選考
- 選考カンバン
- 応募・選考詳細の関連面談 / 関連タスク
- 転職サービス詳細、会社詳細の関連表示

## データ方針

デモデータは架空の情報だけを使います。

- 実在する応募先、担当者、メールアドレス、電話番号は使わない
- 会社名、サービス名、担当者名はデモ用と分かる名前にする
- スクリーンショットで見たときに、転職活動の流れが伝わる量にする
- ダッシュボードが空にならないよう、期限・面談・優先度を意図的に分散する

## 作成されるデータ

`supabase/seed.sql` では、以下のデータを作成します。

| 種別 | 件数 | 内容 |
| --- | ---: | --- |
| 転職サービス | 4 | エージェント、スカウト、転職サイト、SNS |
| 会社 | 5 | SaaS、AI、FinTech、Cloud、HealthTech系 |
| 担当者 | 5 | エージェント、企業人事、スカウト担当 |
| 求人 | 8 | Frontend、Backend、AI Productなど |
| 応募・選考 | 8 | 気になる、応募予定、応募済み、面接、内定など |
| 面談予定 | 5 | 直近予定、一次面接、最終面接など |
| タスク | 12 | 期限切れ、今日、7日以内、完了済みを含む |

## 事前準備

Supabase Authenticationで、デモデータを紐づけるユーザーを1人用意します。

そのユーザーの `id` を確認してください。

確認場所:

```text
Supabase Dashboard
Authentication
Users
対象ユーザー
User UID
```

## 実行手順

1. `supabase/seed.sql` を開きます。
2. 先頭付近の `seed_user` にあるUUIDを、対象ユーザーのUser UIDへ置き換えます。

```sql
select '00000000-0000-0000-0000-000000000000'::uuid as user_id
```

例:

```sql
select '12345678-1234-1234-1234-123456789abc'::uuid as user_id
```

3. Supabase SQL Editorに `supabase/seed.sql` のSQL本文を貼り付けます。
4. SQLを実行します。
5. Career Trackerへ対象ユーザーでログインします。
6. 各画面でデモデータが表示されることを確認します。

## 再実行について

`supabase/seed.sql` は再実行できます。

同じユーザーに対して実行すると、以下のデモデータを削除してから再作成します。

- `Demo` で始まる転職サービス
- `Demo` で始まる会社
- `Demo` で始まる担当者
- `Demo` で始まる求人

関連する応募・選考、面談予定、タスクは外部キー制約により合わせて削除されます。

注意:

- デモデータ以外の手入力データは削除しない設計です。
- ただし、同じ命名規則で `Demo` から始まるデータを手入力している場合は削除対象になります。

## 実行後の確認画面

まず以下を確認してください。

1. `/dashboard`
2. `/jobs`
3. `/applications`
4. `/applications/kanban`
5. `/tasks`
6. `/interviews`

スクリーンショット撮影では、以下を優先します。

1. ダッシュボード
2. 選考カンバン
3. 応募・選考詳細
4. 求人一覧
5. タスク・期限一覧

## 注意点

- 本番用Supabaseプロジェクトで実行する場合は、事前に対象ユーザーとデータ内容を確認してください。
- 実データが入っている環境では、誤って別ユーザーIDを指定しないよう注意してください。
- Service Role Keyは不要です。SQL Editorから実行してください。
